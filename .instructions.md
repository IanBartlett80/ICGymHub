# GymHub Development Instructions for AI Assistants

> **Context:** GymHub is a live production SaaS application for gymnastics club management, currently serving real customers with active subscriptions. This document provides critical guidelines for AI assistants working on this codebase.

---

## 🔒 1. Production-Safety Requirements

**GymHub is a LIVE PRODUCTION application.**

- Backend, API, and database are currently serving real customers
- All code changes must be treated as **high-risk** unless proven otherwise
- Real users are actively using the system 24/7

### Your Top Priority:
✅ **DO NOT** generate or suggest anything that could break existing functionality  
✅ When unsure, you **MUST ASK** for confirmation before generating risky modifications  
✅ Assume defensive posture: preservation over optimization

---

## 🗄️ 2. Database Safety Rules

**Production Database:** PostgreSQL hosted on DigitalOcean  
**Risk Level:** CRITICAL

### Prohibited Operations (Never Generate These):
- ❌ `DROP TABLE` or `DROP COLUMN`
- ❌ `TRUNCATE` statements
- ❌ Direct data overwrites without backups
- ❌ Irreversible schema changes
- ❌ Removing foreign keys or constraints without migration plan
- ❌ Changing column types that could cause data loss

### Allowed Operations (Safe, Additive Changes):
- ✅ Creating new tables
- ✅ Adding new columns with safe defaults (`DEFAULT NULL` or sensible values)
- ✅ Creating new indexes (non-blocking when possible)
- ✅ Adding foreign keys with proper cascades
- ✅ Prisma migrations that maintain existing data

### Required Workflow for Database Changes:

1. **ALWAYS ASK FIRST:**
   - "Does this task require database schema changes?"
   - "Is this for development, staging, or production?"
   - "Do you want me to generate a Prisma migration or raw SQL?"

2. **For Prisma Schema Changes:**
   - Propose the schema change
   - Wait for approval
   - Generate migration with `prisma migrate dev --name descriptive_name`
   - Never run `prisma migrate deploy` to production without explicit approval

3. **For Direct Database Queries:**
   - Request connection details only when absolutely necessary
   - Never assume credentials
   - Always use parameterized queries to prevent SQL injection

## 🧪 3. Code Change Safety Rules

### Before Generating Code:

1. **Request Context First:**
   - "Please share the relevant file(s) so I can see the current implementation"
   - "What is the expected behavior currently?"
   - "Are there tests covering this functionality?"

2. **Preserve Existing Logic:**
   - Never delete or refactor code unless explicitly requested
   - Assume all existing code serves a purpose
   - Flag any breaking changes with clear warnings

3. **Risk Assessment:**
   Ask yourself: Could this change cause...
   - Breaking API contracts? (response shape changes, endpoint removals)
   - Auth/session disruptions? (login failures, permission issues)
   - Data loss? (unvalidated deletes, cascade issues)
   - Downtime? (crashes, infinite loops, memory leaks)
   - Regression? (loss of existing features)

### Code Generation Guidelines:

- **Authentication & Authorization:** Never bypass or weaken existing auth checks
- **API Endpoints:** Maintain backward compatibility; don't change response structures
- **Database Queries:** Use Prisma ORM; avoid raw SQL unless necessary
- **Error Handling:** Always include try-catch blocks and proper error messages
- **Validation:** Maintain or strengthen input validation, never weaken it

## 📦 4. Deployment & Environment Awareness

**Production Environment:** DigitalOcean App Platform  
**Database:** Managed PostgreSQL on DigitalOcean  
**Framework:** Next.js 15 (App Router)  
**ORM:** Prisma 5.22.0

### Deployment Checklist Questions:

Before suggesting deployment-related changes:
1. "Is this change intended for development, staging, or production?"
2. "Have you tested this locally?"
3. "Is there a rollback plan if this breaks?"
4. "Do we need a database backup before deploying?"

### Environment Variables:
Never hardcode sensitive values. Always use environment variables:
- `DATABASE_URL` - Database connection string
- `NEXTAUTH_SECRET` - NextAuth session secret
- `JWT_SECRET` - JWT signing secret
- `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_TENANT_ID` - Microsoft Entra ID
- `AZURE_SENDER_EMAIL` - Email sender configuration

## 🧭 5. Session Startup Protocol

**When starting a new session or receiving a new task:**

### Step 1: Scope Confirmation
Ask:
- "What is the goal of this task?" (feature, bug fix, refactor, etc.)
- "Is this for development or production deployment?"
- "What is the expected timeline/urgency?"

### Step 2: Context Gathering
Request:
- Relevant source files
- Database schema (if applicable)
- API endpoint specifications (if applicable)
- Any error messages or logs

### Step 3: Safety Check
Assess:
- Does this touch the database? → Follow Database Safety Rules
- Does this modify authentication? → Extra caution required
- Does this change API contracts? → Backward compatibility check
- Could this affect live users? → Request testing confirmation

### Step 4: Database Connection (If Needed)
**Only ask for database credentials when absolutely necessary:**
- Schema exploration for complex queries
- Testing data migrations
- Debugging production issues

**Never:**
- Store credentials in code
- Log credentials
- Share credentials in plain text responses

## 🚨 7. Common Pitfalls to Avoid

### ❌ Anti-Patterns:
1. **Modifying existing API response shapes** - breaks frontend clients
2. **Changing database column types** - causes data loss or type errors
3. **Removing required fields** - breaks forms and validations
4. **Bypassing authentication** - security vulnerability
5. **Using `any` types** - defeats TypeScript safety
6. **Ignoring error states** - poor user experience
7. **Hardcoding values** - makes code inflexible
8. **Skipping validation** - security and data integrity risks

### ✅ Best Practices:
1. **Additive changes** - add new features without removing old ones
2. **Feature flags** - allow gradual rollouts
3. **Backward compatibility** - support old and new simultaneously
4. **Comprehensive error handling** - try-catch with user-friendly messages
5. **TypeScript strict mode** - no `any`, proper type definitions
6. **Prisma for queries** - avoid raw SQL injection risks
7. **Environment variables** - all config externalized
8. **Testing before deployment** - validate locally first

## 🔐 10. Security Considerations

### Always Maintain:
- Authentication on all protected routes
- Authorization checks (role-based, club-scoped)
- Input validation and sanitization
- SQL injection prevention (use Prisma)
- XSS prevention (proper escaping)
- CSRF protection (NextAuth handles this)

### Never:
- Expose sensitive data in logs or responses
- Store passwords in plain text
- Skip authentication checks
- Trust client-side data without validation
- Hardcode secrets or API keys
